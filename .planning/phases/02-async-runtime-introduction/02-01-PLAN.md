<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Startup error → stderr | Runtime build failure message written to stderr before tracing is initialized |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | Startup error message on stderr | accept | Message is "failed to build tokio runtime: {e}" — the OS error type (e.g. "resource temporarily unavailable") contains no secrets, no PII, no credentials. Acceptable. |
| T-02-02 | Denial of Service | Runtime build failure → process exit | accept | Exit with non-zero code is the correct fail-closed behavior per D-06. Alternative (continue without runtime) is not available since Floem needs the context. Clean exit is better than undefined state. |
| T-02-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed this phase. tokio 1.52.3 is a pre-existing workspace dependency (Package Legitimacy Audit in 02-RESEARCH.md confirms it). No checkpoint required. |

**Threat surface assessment:** This phase introduces no new external input, no network endpoints, no file parsing, and no authentication paths. The runtime is ambient and unused. The only observable external effect is a possible stderr message on catastrophic startup failure. ASVS L1 controls do not add requirements here beyond the fail-closed exit already mandated by D-06.
</threat_model>
